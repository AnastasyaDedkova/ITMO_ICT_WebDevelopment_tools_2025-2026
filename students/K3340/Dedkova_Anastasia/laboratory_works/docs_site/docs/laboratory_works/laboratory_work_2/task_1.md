## Задача 1

Различия между threading, multiprocessing и async в Python
Задача: Напишите три различных программы на Python, использующие каждый из подходов: threading, multiprocessing и async. Каждая программа должна решать считать сумму всех чисел от 1 до 10000000000000. Разделите вычисления на несколько параллельных задач для ускорения выполнения.

В рамках работы были рассмотрены два режима:

- **Режим A** - полный диапазон по условию задания: `1 .. 10_000_000_000_000`
- **Режим B** - уменьшенный диапазон для практического тестирования и сравнения производительности



## Логика решения

### Общая идея

Во всех трёх программах использовалась одинаковая схема:

1. Диапазон от `1` до `N` разбивается на несколько частей.
2. Для каждой части создаётся отдельная задача:
   - поток для `threading`
   - процесс для `multiprocessing`
   - асинхронная задача для `asyncio`
3. Каждая задача вычисляет сумму своей части диапазона.
4. После завершения всех задач частичные суммы складываются в общий результат.
5. Измеряется время выполнения.

Такой подход позволяет сравнить разные модели параллелизма в Python на одной и той же задаче.



## Описание реализованных подходов

### Threading

В файле `task1_threading.py` задача решается с использованием модуля `threading`.

Реализация состоит из следующих основных частей:

- `calculate_sum(start, end, results, index)` — функция, которая вычисляет сумму чисел в выделенном диапазоне от `start` до `end` включительно. Полученная частичная сумма сохраняется в общий список `results` по индексу `index`.
- `split_range(max_number, parts)` — функция, которая разбивает общий диапазон `1..N` на несколько поддиапазонов. Количество частей задаётся числом потоков.
- `main()` — основная функция программы. Она:
  1. выводит параметры запуска;
  2. создаёт список для хранения частичных сумм;
  3. вызывает `split_range(...)` для разбиения диапазона;
  4. создаёт потоки `threading.Thread(...)`;
  5. запускает их через `start()`;
  6. ожидает завершения через `join()`;
  7. суммирует частичные результаты;
  8. измеряет и выводит время выполнения.

Логика работы состоит в том, что общий диапазон разбивается на несколько частей, каждая часть передаётся отдельному потоку, а после завершения всех потоков частичные суммы складываются в итоговый результат.

Особенность этого подхода заключается в том, что он плохо подходит для вычислительных (`CPU-bound`) задач в Python. Причина — наличие `GIL` (Global Interpreter Lock), который не позволяет нескольким потокам одновременно эффективно выполнять Python-байткод на нескольких ядрах.

Поэтому, несмотря на формальное разбиение задачи на несколько потоков, существенного ускорения для данной вычислительной задачи не наблюдается.


### Multiprocessing

В файле `task1_multiprocessing.py` задача решается с использованием модуля `multiprocessing`.

Реализация состоит из следующих основных частей:

- `calculate_sum(start, end, queue)` — функция, которая вычисляет сумму чисел в выделенном диапазоне от `start` до `end` включительно и помещает результат в очередь `queue`.
- `split_range(max_number, parts)` — функция разбиения диапазона `1..N` на несколько поддиапазонов.
- `main()` — основная функция программы. Она:
  1. выводит параметры запуска;
  2. разбивает диапазон на части;
  3. создаёт очередь `multiprocessing.Queue()` для передачи частичных результатов;
  4. создаёт процессы `multiprocessing.Process(...)`;
  5. запускает их;
  6. считывает частичные суммы из очереди;
  7. ожидает завершения всех процессов через `join()`;
  8. вычисляет итоговую сумму и время выполнения.

В этой реализации каждая часть диапазона обрабатывается отдельным процессом. В отличие от потоков, процессы имеют собственное адресное пространство и собственный интерпретатор Python, поэтому могут реально выполняться параллельно на нескольких ядрах процессора.

Именно этот подход лучше всего подходит для тяжёлых вычислительных задач и показал наилучший результат в эксперименте.


### Asyncio

В файле `task1_async.py` задача решается с использованием асинхронного подхода на базе `asyncio`.

Реализация включает следующие основные части:

- `calculate_sum(start, end)` — асинхронная функция, которая вычисляет сумму чисел в диапазоне от `start` до `end` включительно и возвращает частичный результат.
- `split_range(max_number, parts)` — функция разбиения общего диапазона `1..N` на несколько поддиапазонов.
- `main()` — асинхронная основная функция программы. Она:
  1. выводит параметры запуска;
  2. разбивает диапазон на части;
  3. создаёт список асинхронных задач;
  4. запускает их одновременно через `asyncio.gather(...)`;
  5. собирает частичные суммы;
  6. вычисляет итоговую сумму и время выполнения.

Формально задача разбивается на несколько асинхронных подзадач, однако сама работа остаётся вычислительной (`CPU-bound`). Внутри функции нет операций ожидания, таких как сетевые запросы, чтение файлов или обращение к базе данных, поэтому асинхронность не может дать существенного выигрыша.

Таким образом, реализация с `asyncio` была выполнена для сравнения и демонстрации того, что асинхронность хорошо подходит для `I/O-bound` задач, но не является эффективным инструментом для тяжёлых CPU-bound вычислений.



## Почему был выбран режим B

Полный диапазон по условию задания составляет:

`1 .. 10_000_000_000_000`

Полный практический запуск такой задачи для всех трёх подходов оказался неудобен.  
Для `threading` и `asyncio` предварительные измерения показали, что время выполнения на полном диапазоне может составлять несколько суток. Поэтому для корректного сравнительного эксперимента был введён тестовый режим:

`B = 5_000_000_000`

Этот диапазон был выбран по следующим причинам:

1. Он достаточно большой, чтобы различия между подходами были хорошо заметны.
2. Его ещё можно выполнить за разумное время на обычном ноутбуке.
3. Он позволяет получить реальные измерения времени выполнения.
4. На меньшем диапазоне (`100_000_000`) все программы выполнялись слишком быстро, и различия между подходами проявлялись слабее.

В работе использовались три уровня диапазона:

- `100_000_000` — проверочный запуск для проверки корректности работы программ;
- `5_000_000_000` — основной сравнительный эксперимент;
- `10_000_000_000_000` — полный диапазон из условия задания, для которого строилась теоретическая оценка времени выполнения.

### Формула расчёта теоретического времени

Для оценки времени выполнения на полном диапазоне использовалась линейная экстраполяция:

T_A = T_B * (N_A / N_B)

где:
- T_A — оценочное время на полном диапазоне;
- T_B — реально измеренное время на диапазоне B;
- N_A — полный диапазон;
- N_B — диапазон режима B.

В данной работе:

- N_A = 10 000 000 000 000
- N_B = 5 000 000 000

Следовательно, коэффициент масштабирования равен:

k = N_A / N_B = 10 000 000 000 000 / 5 000 000 000 = 2000


То есть оценка времени для полного диапазона вычислялась по формуле:

T_A = T_B * 2000

### Оценка погрешности

Данная формула даёт приближённую оценку, а не точное значение.  
Она основана на предположении, что время выполнения растёт примерно линейно с ростом числа операций.

Источники возможной погрешности:

- накладные расходы на создание потоков, процессов и асинхронных задач;
- влияние планировщика операционной системы;
- изменение нагрузки на ноутбук во время длительного выполнения;
- особенности кэширования и распределения нагрузки между ядрами;
- возможное снижение производительности при длительном непрерывном вычислении.

Следовательно, теоретические значения для диапазона `A` следует рассматривать как оценочные.  
Наиболее корректными для сравнения являются реальные результаты, полученные на диапазоне `B = 5_000_000_000`.

## Используемые математические формулы

### Формула суммы чисел от 1 до N

Для проверки корректности результатов использовалась формула суммы арифметической прогрессии:

\[
S = \frac{N(N+1)}{2}
\]

где:
- \(S\) — сумма чисел от `1` до `N`;
- \(N\) — верхняя граница диапазона.

Например, для полного диапазона из задания:

\[
S = \frac{10\,000\,000\,000\,000 \cdot 10\,000\,000\,000\,001}{2}
\]

\[
S = 50\,000\,000\,000\,005\,000\,000\,000\,000
\]

### Формула теоретической оценки времени

Для оценки времени выполнения на полном диапазоне использовалась формула линейного масштабирования:

\[
T_A = T_B \cdot \frac{N_A}{N_B}
\]

В данной работе:

\[
T_A = T_B \cdot 2000
\]

где:
- \(T_B\) — реально измеренное время на диапазоне `5_000_000_000`;
- \(T_A\) — оценка времени на диапазоне `10_000_000_000_000`.

### Почему `threading` показывает слабый результат

Потоки в Python ограничены `GIL`, поэтому для CPU-bound задач они не обеспечивают полноценного параллельного выполнения Python-байткода на нескольких ядрах.  
Из-за этого `threading` в подобных задачах не даёт настоящего масштабируемого ускорения, а рост числа потоков быстро перестаёт приносить заметную пользу.

Это подтверждается дополнительным экспериментом:

- `threading`, 1 worker — `274.764` сек
- `threading`, 16 workers — `192.763` сек

То есть некоторое уменьшение времени есть, но оно значительно слабее, чем у `multiprocessing`, и не означает, что потоки являются эффективным решением для тяжёлых вычислительных задач в Python.

### Почему `asyncio` не даёт заметной разницы

`asyncio` в данной задаче не является подходящим инструментом, поскольку рассматриваемая нагрузка является вычислительной (`CPU-bound`), а не связанной с ожиданием (`I/O-bound`).  
Асинхронный подход эффективен тогда, когда внутри задач есть операции ожидания: сетевые запросы, работа с файлами, обращение к базе данных и другие задержки ввода-вывода. В данной лабораторной таких операций внутри вычисления суммы нет.

Следовательно, `asyncio` здесь не даёт реального параллельного ускорения вычислений, а лишь добавляет накладные расходы на организацию асинхронных задач и переключение между ними.

Это подтверждается экспериментом:

- `async`, 1 worker — `285.872` сек
- `async`, 16 workers — `190.771` сек

Улучшение есть, но оно значительно слабее, чем у `multiprocessing`.

### Почему `multiprocessing` оказался лучшим

Процессы работают независимо и могут выполняться параллельно на разных ядрах процессора.  
Из-за этого `multiprocessing` лучше всего подходит для CPU-bound вычислений.

Дополнительный эксперимент по workers показывает хорошую масштабируемость:

- `multiprocessing`, 1 worker — `293.219` сек
- `multiprocessing`, 2 workers — `138.952` сек
- `multiprocessing`, 4 workers — `72.744` сек
- `multiprocessing`, 8 workers — `49.125` сек
- `multiprocessing`, 16 workers — `37.895` сек

То есть при увеличении числа процессов производительность действительно растёт.

### Анализ теоретической оценки для полного диапазона

Линейная экстраполяция показала, что для диапазона `1..10_000_000_000_000` время выполнения составило бы примерно:

- `threading` — `5.09` суток
- `async` — `4.79` суток
- `multiprocessing` — `1.56` суток

Эти данные подтверждают, что практический запуск полного диапазона для `threading` и `asyncio` малореален, а `multiprocessing` остаётся единственным относительно оправданным вариантом.

## Вывод

В ходе выполнения лабораторной работы были реализованы три подхода к организации параллельного выполнения вычислений в Python:

- `threading`
- `multiprocessing`
- `asyncio`

Во всех трёх случаях использовалась одинаковая схема решения: диапазон чисел разбивался на части, для каждой части создавалась отдельная задача, после чего частичные суммы объединялись в итоговый результат.

В ходе работы были выполнены:

1. проверочный запуск на диапазоне `1..100_000_000`;
2. основной сравнительный эксперимент на диапазоне `1..5_000_000_000`;
3. дополнительный эксперимент по изменению числа workers (`1, 2, 4, 8, 16`);
4. теоретическая оценка времени выполнения на полном диапазоне `1..10_000_000_000_000`.

Полученные результаты показали, что:

- `threading` слабо подходит для CPU-bound задач, так как из-за `GIL` не обеспечивает полноценного параллельного выполнения вычислений на нескольких ядрах;
- `asyncio` не является подходящим инструментом для тяжёлых вычислений, поскольку предназначен прежде всего для `I/O-bound` задач и в данной работе не даёт реального вычислительного ускорения;
- `multiprocessing` является наиболее эффективным подходом для вычислительных задач, поскольку процессы могут действительно выполняться параллельно на нескольких ядрах процессора.

Дополнительный эксперимент с различным числом workers подтвердил, что именно `multiprocessing` лучше всего масштабируется при увеличении числа рабочих единиц.
