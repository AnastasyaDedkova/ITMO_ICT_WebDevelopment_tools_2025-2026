# Лабораторная работа №2  
## Задача 2. Параллельный парсинг веб-страниц с сохранением в базу данных

## Постановка задачи
Задача: Напишите программу на Python для параллельного парсинга нескольких веб-страниц с сохранением данных в базу данных с использованием подходов threading, multiprocessing и async. Каждая программа должна парсить информацию с нескольких веб-сайтов, сохранять их в базу данных.
В качестве источника данных использовались страницы книг сайта **Project Gutenberg**.  
В качестве хранилища данных использовалась база данных из первой лабораторной работы.


## Используемые технологии
- Python 3
- `requests`
- `aiohttp`
- `threading`
- `multiprocessing`
- `asyncio`
- `BeautifulSoup`
- `SQLModel`
- PostgreSQL



## Структура решения

В папке `task2` используются следующие файлы:

- `collect_book_urls.py` - сбор ссылок на книги и сохранение их в `book_urls.txt`
- `parser_utils.py` - общая логика:
  - загрузка URL;
  - парсинг страницы;
  - сохранение книги в БД;
  - запись результатов в CSV;
  - очистка импортированных записей
- `task2_threading.py` - реализация парсинга через потоки
- `task2_multiprocessing.py` - реализация парсинга через процессы
- `task2_async.py` - реализация парсинга через асинхронные запросы
- `run_task2_all.py` - общий запуск всех методов
- `task2_preview.py` - тестовый режим без БД, с записью результатов в JSONL
- `task2_db_results.csv` - результаты замеров времени для работы с БД
- `task2_preview_results.csv` - результаты замеров времени без БД
- `book_urls.txt` - список ссылок для обработки



## Логика работы программы

### 1. Сбор ссылок
Сначала запускается файл `collect_book_urls.py`.

Программа:
- запрашивает у пользователя количество ссылок;
- собирает ссылки на книги с нескольких страниц Project Gutenberg;
- сохраняет найденные URL в `book_urls.txt`.

Таким образом формируется единый набор входных данных для всех трёх методов.



### 2. Тестовый режим без БД
Для предварительной проверки качества парсинга используется файл `task2_preview.py`.

В этом режиме:
- ссылки загружаются из `book_urls.txt`;
- данные парсятся без записи в БД;
- результаты сохраняются в JSONL-файлы;
- отдельно замеряется время выполнения.

Этот этап нужен для проверки:
- корректности парсинга,
- правильности извлечения названия, автора, описания и года,
- работоспособности всех трёх подходов до подключения базы данных.



### 3. Основной режим по заданию
Основной эксперимент выполняется с сохранением в PostgreSQL.

Для каждого метода:
- берётся один и тот же набор ссылок из `book_urls.txt`;
- выполняется полный цикл:
  1. загрузка страницы;
  2. парсинг HTML;
  3. сохранение результата в БД;
- измеряется **полное время выполнения программы**.

Перед запуском следующего метода база данных очищается от предыдущего импорта парсера, чтобы все три подхода работали в одинаковых условиях.  
Это необходимо для корректного сравнения.



## Какие данные сохраняются в базу данных

Из страницы книги извлекаются:
- `title` - название книги
- `author` - автор
- `publication_year` - год публикации
- `description` - описание

При сохранении в БД:
- создаётся запись в таблице `book`;
- создаётся или используется существующий жанр `Project Gutenberg`;
- создаётся связь в таблице `bookgenrelink`.

Таким образом, данные из парсера встраиваются в уже существующую структуру базы данных из первой лабораторной работы.



## Реализация подходов

### 1. Threading
В реализации на основе потоков список URL делится на несколько частей.  
Каждый поток обрабатывает свой фрагмент списка.

Особенности:
- хорошо подходит для I/O-bound задач;
- относительно простая реализация;
- ускоряет выполнение по сравнению с последовательным режимом.



### 2. Multiprocessing
В реализации на основе процессов список URL также делится на части, но каждая часть обрабатывается отдельным процессом.

Особенности:
- использует несколько процессов вместо потоков;
- подходит для независимых задач;
- требует немного большего накладного ресурса, чем `threading`.



### 3. Async
В реализации на основе `asyncio` и `aiohttp` запросы выполняются асинхронно.

Особенности:
- не требует создания отдельных потоков или процессов для каждой задачи;
- особенно эффективен для большого числа сетевых запросов;
- показал наилучшее время выполнения.

## Анализ результатов

В эксперименте использовался одинаковый набор из **950 ссылок**, поэтому сравнение методов можно считать корректным.

По полученным результатам можно сделать следующие выводы:

1. **Async** показал наилучший результат — **56.8838 сек**.  
   При большом количестве сетевых запросов асинхронный подход оказался самым эффективным, так как позволяет обрабатывать множество операций ввода-вывода без создания большого числа потоков и процессов.

2. **Threading** показал результат **244.7244 сек**.  
   Этот подход также позволяет распараллелить выполнение I/O-bound задач, однако при большом объёме запросов уступает асинхронной реализации.

3. **Multiprocessing** показал результат **268.7236 сек**.  
   Несмотря на параллельное выполнение, данный подход оказался самым медленным из трёх рассматриваемых методов. Для задачи парсинга веб-страниц с записью в базу данных использование отдельных процессов создаёт дополнительные накладные расходы, которые не дают преимущества по сравнению с потоками и особенно с async.

4. На объёме **950 URL-адресов** различия между подходами проявились достаточно явно.  
   Это позволяет сделать более надёжный вывод о том, что для массового парсинга веб-страниц с сохранением результатов в БД наиболее подходящим вариантом является асинхронный подход.
